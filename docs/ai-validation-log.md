# AI Validation Log

AI accelerates drafting in this repo; judgment stays human. Every AI-generated
case is checked against the application's real behavior before it enters the
suite. This log records where the AI got something wrong and how I corrected it.
The point is the differentiator: directing AI and catching its mistakes is the
skill, not simply using it.

| Date | Area / file | What the AI got wrong | The correction | Lesson |
|------|-------------|-----------------------|----------------|--------|
| YYYY-MM-DD | `src/fixtures/auth.ts`, security cases | Drafted helpers that logged in as a hardcoded seeded account (`jim@juice-sh.op`) and assumed stable seeded IDs. | Switched to registering a fresh, unique user per test via `POST /api/Users`, so nothing depends on a seeded password that can change between Juice Shop versions. | Seeded credentials are a hidden coupling; per-test data is more robust and reads better. |
| YYYY-MM-DD | `tests/security/chatbot-prompt-injection.spec.ts` | Used `page.waitForTimeout(1500)` to pace chatbot turns. | Replaced fixed sleeps with a wait on the chatbot's response, so the case is deterministic rather than timing-dependent. | Fixed sleeps are flake waiting to happen; wait on real signals. |
| YYYY-MM-DD | `docker-compose.yml` / db layer | Suggested bind-mounting Juice Shop's data directory to read the SQLite file. | Rejected: the mount shadows seed files the app needs at boot. The db fixture copies the file out with `docker compose cp` instead. | Verify a proposed approach against how the app actually starts, not just whether it compiles. |
| 2026-06-28 | `docker-compose.yml` healthcheck | Wrote the healthcheck as `["CMD-SHELL", "node -e …"]`, assuming a shell and `node` on `PATH`. The pinned v20 image is distroless (entrypoint `/nodejs/bin/node`, no `/bin/sh`, `node` not on `PATH`), so Docker's `/bin/sh -c` could never start it — the container stayed `unhealthy` forever and `docker compose up --wait` exited 1, locally and in all three app-dependent CI jobs. | Switched to exec form with the absolute interpreter path: `["CMD", "/nodejs/bin/node", "-e", …]` — no shell, no `PATH` lookup. `--wait` now reports healthy in ~6s. | Confirm container-level commands against the actual image; a distroless base has no shell, and a healthcheck that can never pass silently breaks every `--wait`. |
| 2026-06-28 | `src/fixtures/db.ts` | Loaded the SQLite driver with a static `import { DatabaseSync } from 'node:sqlite'` (then tried a runtime `require`); both crashed Playwright's TS loader with `Expected … "source" from the "load" hook … got null`, because Playwright routes every import/require through its own loader and chokes on the experimental builtin. | Loaded it with `process.getBuiltinModule('node:sqlite')`, which fetches the builtin directly and bypasses the loader; kept full types via `import type` + a cast. Verified the fix in Node's v24 docs and with a green run. | "Works under bare `node`" ≠ "works under the test runner's loader"; for experimental builtins, `process.getBuiltinModule` sidesteps the toolchain. |
| 2026-06-28 | `src/pages/LoginPage.ts`, `src/pages/BasePage.ts` | The first-draft login POM used accessible-name substrings that matched more than one node on the real v20 page: `getByLabel(/password/i)` resolved to **2 elements** (the password `<input>` *and* the "Button to display the password" show/hide toggle) → strict-mode violation; the submit `getByRole('button', { name: /log\s?in/i })` also matched "Login with Google" (which contains the substring "Login"); and the cookie-banner dismiss was drafted as a `link` named "Me want it!" when it is a `button` whose accessible name is "dismiss cookie message". (The pre-existing `loginViaUi` in `auth.ts` carries the same latent gaps — never run against the UI.) | Disambiguated by role + exact name, read straight off the ARIA snapshot Playwright captures on failure: `getByRole('textbox', { name: /password/i })` (the toggle is a button, not a textbox), `getByRole('button', { name: 'Login', exact: true })`, and `getByRole('button', { name: /dismiss cookie message/i })`. | An `aria-label` substring is not a unique locator: form inputs ship sibling action buttons, and "Login" is a prefix of "Login with Google". Disambiguate by role + `exact`, and let the failure's ARIA snapshot tell you the real accessible names instead of guessing. |
| 2026-06-28 | `tests/security/dom-xss-search.spec.ts` | The documented reference snippet (and my first draft from it) wrote the (B) target-state test as a bare `toHaveCount(0)` immediately after `page.goto('/#/search?q=…')`. Against the Angular SPA this races the render: `goto` resolves on `load` while the page momentarily has **zero** iframes, so the absence assertion was satisfied *before* the payload was injected. `test.fail()` then reported an **unexpected pass** — a false "the XSS is fixed" signal against the still-vulnerable app. Caught by running the spec live (not by reading it). | Anchored the assertion on a positive render signal first — `await expect(page.locator('#searchValue')).toBeAttached()` (the span that holds the reflected term, confirmed on v20) — then `toHaveCount(0)`. Verified deterministic with `--repeat-each=5` (10/10: 5× (A) pass, 5× (B) expected-fail). | Asserting the *absence* of something a SPA renders asynchronously is racy by construction: wait for a positive signal that the region has rendered before checking the thing isn't in it. A passing `test.fail()` review-clean snippet can still be wrong — run it. |
| 2026-06-28 | `tests/security/error-leakage.spec.ts` | The documented case (and the prompt) said sending a single quote `'` as `q` to `/rest/products/search` provokes a `>=500` error leaking `SQLITE_ERROR` **and** a `SELECT ... FROM Products` fragment. On v20, `'` actually returns **200** with an empty result set — no error — and even on a genuine error the full `SELECT ... FROM Products` query text is **not** echoed; only the engine error reaches the client. So the (A) assertion would never match and the (B) `test.fail()` would report a false "it's fixed" pass. | Probed payloads through the test's own `request.newContext` channel: `')` (closes the LIKE literal and unbalances the WHERE parens) reliably returns **500 `text/html`** whose title/heading carry `SQLITE_ERROR: near ")": syntax error`. Switched the payload to `')`, asserted the observed engine token `/SQLITE_ERROR/i` plus `status 500` + `text/html`, and dropped the never-present `SELECT ... FROM Products` alternative. | A payload that "should" error can be silently swallowed by a given version; confirm the exact input that triggers the leak and the exact tokens that leak against the running app — don't trust the doc's example wording. |
